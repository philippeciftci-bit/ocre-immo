# Conception

**Avant (M/37 → M/42)** : le critère cas A vérifiait la présence du cookie `ocre_jwt` envoyé par le navigateur dans le fetch POST `/api/login.php`. Critère **fragile** : Safari iPad ITP bloque ce cookie sur cross-subdomain fetch, même avec `SameSite=None` (sur certaines configs iOS 17+ "Prevent Cross-Site Tracking").

**Après (M/43)** : le critère cas A devient **purement serveur** = un timestamp DB.
- `max(auth_users.last_login_at, auth_users.last_magic_link_consumed_at)` dans la fenêtre `[now - magic_link_ttl_hours × 3600, now]` → cas A.
- Plus de dépendance au cookie navigateur. Funcione même sur Safari iPad sans cookies.

Source de vérité : **DB**, pas client.

# Migration DB (idempotente)

`auth-root/migrations/M_AUTH_FLOW_CAS_A_TTL.sql` :
```sql
ALTER TABLE auth_users ADD COLUMN IF NOT EXISTS last_magic_link_consumed_at DATETIME NULL DEFAULT NULL;
ALTER TABLE auth_users ADD INDEX IF NOT EXISTS idx_email_last_access (email, last_login_at, last_magic_link_consumed_at);
```
Exécutée en prod début de mission.

# Diffs

## `auth-root/api/magic-link/validate.php`
Stamper aussi `last_magic_link_consumed_at` au consume :
```diff
- $db->prepare("UPDATE auth_users SET last_login_at = NOW() WHERE id = ?")->execute([$userId]);
+ $db->prepare("UPDATE auth_users SET last_login_at = NOW(), last_magic_link_consumed_at = NOW() WHERE id = ?")->execute([$userId]);
```

## `auth-root/api/login.php`
- Lookup `last_login_at` + `last_magic_link_consumed_at` dans la query user.
- Remplacement du check cookie-based par check TTL :
```php
$lastA = $user['last_login_at'] ? strtotime($user['last_login_at']) : 0;
$lastB = $user['last_magic_link_consumed_at'] ? strtotime($user['last_magic_link_consumed_at']) : 0;
$lastAccess = max($lastA, $lastB);
$ttlSeconds = $ttlHours * 3600;
$withinTtl = $lastAccess > 0 && (time() - $lastAccess) < $ttlSeconds;
```
- Dans le bloc cas A : **re-crée session SSO inline** (nouveau JWT + INSERT auth_sessions + `auth_set_cookies`) puis resolve slug tenant + return `action=direct`.
- `redirect_url` query param = `?_s=<sso>&source=ttl_login` (ou `?source=ttl_login` selon provisioning).

# Sécurité

Le cas A ne dépend plus du cookie navigateur, mais ces mitigations sont intactes :
- JWT signature HMAC (création + validation sur me.php).
- `auth_sessions` jti lookup pour valider la session est active et non révoquée.
- CORS Origin whitelist stricte sur `/api/login.php`.
- Tenant `redirect_url` basé sur `users.slug` DB (pas d'input arbitraire).
- **Rate limit `login_unified` 30/h par IP** déjà en place : empêche un attaquant de tester des emails à grande échelle.

Risque résiduel : si un attaquant connaît l'email d'un user qui s'est loggé récemment, il peut déclencher une réponse `action=direct` avec `redirect_url`. **MAIS** il n'aura pas le SSO token / cookie `ocre_jwt` (car ils sont posés en response et nécessitent le navigateur du user attaquant pour être utiles). Le redirect_url SEUL n'autorise pas l'accès au tenant. → Pas de risque exploitable.
