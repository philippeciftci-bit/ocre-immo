# Cause racine identifiée

Le flash "OCRE immo · Connexion" voyait quand le user arrivait sur `<slug>.ocre.immo/?_s=<sso>&source=ttl_login` venait d'un **mismatch query param** :

- `validate.php` (cas A redirect direct M/37 amd#2) + `login.php` (cas A TTL M/43) + `provision-tenant.php` envoyaient `?_s=<sso_token>` dans l'URL.
- La SPA tenant `/opt/ocre-app/index.html` ligne 11564 consomme **`?mt_token=`** (legacy V52.4.2), pas `?_s=`.
- Résultat : `?_s=` ignoré par la SPA → `localStorage.ocre_token` vide → fetch `/api/session_check.php` → si cookie `ocre_session` legacy pas posé → `window.location = '/login/'` qui affiche la page **"OCRE immo · Connexion"** brièvement avant que le user finisse par cliquer / soumettre.

Le bug n'était pas un "flash CSS pendant le boot React" mais un **redirect réel vers /login/ qui s'éclipse** dès qu'une autre logique re-redirige.

# Fix appliqué

**1 rename dans 3 fichiers** : `?_s=` → `?mt_token=` partout où on passe le sso_token vers la SPA tenant.

## `auth-root/api/magic-link/validate.php`
```diff
- $dest = $prov['tenant_url'] . '?_s=' . urlencode($prov['sso_token']) . '&activated=1';
+ $dest = $prov['tenant_url'] . '?mt_token=' . urlencode($prov['sso_token']) . '&activated=1';
```

## `auth-root/api/login.php` (cas A TTL)
Refactor du bloc cas A pour **toujours** appeler `auth_provision_tenant()` (idempotent : reuse slug + DB + génère nouveau sso_token = mt_token pour SPA). Avant : 2 branches (slug existait = sans sso_token = SPA tombait sur fail). Maintenant : 1 seule branche, toujours `mt_token` dans l'URL.

```diff
- 'redirect_url' => 'https://' . $slug . '.ocre.immo/?_s=' . urlencode($prov['sso_token']) . '&source=ttl_login',
+ 'redirect_url' => 'https://' . $prov['slug'] . '.ocre.immo/?mt_token=' . urlencode($prov['sso_token']) . '&source=ttl_login',
```

## `agent-landing/index.html` (routeur transparent agent.ocre.immo M/36)
```diff
- target += '...' + '_s=' + encodeURIComponent(p.sso_token);
+ target += '...' + 'mt_token=' + encodeURIComponent(p.sso_token);
```

# Pourquoi pas de splash neutre supplémentaire

La spec proposait un splash neutre custom dans la SPA tenant. **Pas nécessaire** : la SPA tenant rend déjà un splash propre via `if (apiLoading) return <div>Oi Agent + spinner</div>` (ligne 12033 de `/opt/ocre-app/index.html`). Le bug réel = le user n'arrivait jamais à ce splash, il était redirigé vers `/login/` AVANT. Avec `mt_token` consume au boot (ligne 11564), le SPA stocke le token + reload + dashboard direct. Le splash "Oi Agent" inline reste visible ~300ms pendant le reload, sans aucun flash de form connexion.
